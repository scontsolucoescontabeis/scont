Como estruturar a ferramenta?

Escolha a abordagem arquitetural antes de detalhar o design

A

Arquivo único (index.html + script.js + styles.css)

Mesmo padrão do Controle de Frequência. Sidebar troca seções via JavaScript — sem recarregar a página. Lançamentos, Configurações e Escalas ficam em "telas" dentro do mesmo arquivo.

Prós

  • Consistente com as ferramentas existentes
  • Uma única verificação de auth
  • Navegação instantânea entre telas

Contras

  • Arquivo JS pode ficar grande (~1500 linhas)
B

Múltiplos arquivos HTML (uma página por tela)

index.html = Lançamentos, configuracoes.html, escalas.html. Cada página tem seu próprio guard de auth. Sidebar linka entre as páginas.

Prós

  • Cada arquivo é menor e mais focado
  • Mais fácil de manter isoladamente

Contras

  • Estado não persiste entre navegações (recarrega)
  • 3 verificações de auth separadas
  • Mais arquivos para gerenciar
C

Arquivo único com módulos JS separados por tela

index.html único com sidebar, mas o JS de cada tela fica em arquivos separados (lancamentos.js, configuracoes.js, escalas.js) importados dinamicamente. Melhor organização sem perder a fluidez.

Prós

  • Navegação fluida + código organizado
  • Uma auth, estado compartilhado entre telas

Contras

  • Ligeiramente mais complexo de estruturar

Recomendação

Opção A — segue exatamente o padrão do Controle de Frequência que já funciona bem no portal. O JS será dividido em funções bem nomeadas por responsabilidade dentro do script.js, mantendo tudo legível.
